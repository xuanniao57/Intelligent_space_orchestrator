# 项目治理与权限方案

本文档定义智场同语中枢 Agent 的协作角色、权限、审核和合并规则。目标是让更多人能参与，同时保护 `main` 分支、接口协议和真实设备执行链的稳定性。

## 推荐管理结构

### 仓库所有者

仓库所有者拥有最终决策权，负责：

- 邀请或移除合作者。
- 设置 GitHub 权限和分支保护。
- 处理无法达成一致的技术路线、发布窗口和安全问题。
- 审核高风险变更，尤其是现场设备、密钥、第三方代码和协议破坏性改动。

### 维护者

建议邀请 2 到 4 位长期合作者作为维护者，而不是把所有参与者都设为管理员。维护者负责：

- 审核和合并 Pull Request。
- 维护接口规范、运行手册、注册表和核心模块质量。
- 帮助普通贡献者拆分任务、补测试、补文档。
- 在演示或现场部署前确认版本状态。

维护者建议覆盖这些方向：

- 后端与 Agent runtime。
- I/O 协议、语义注册表和设备 ACK。
- 前端监控台与交互。
- 运维、部署和现场联调。

### 模块审核者

模块审核者不一定需要仓库管理权限，但可以作为某个目录的指定 reviewer，例如：

- `central_hub/backend/`
- `central_hub/data/agent_io_registry/`
- `frontend/zhichang_agent_console/`
- `tools/fake_g1/`
- `docs` 和运行手册

### 普通贡献者

普通贡献者通过 fork 或项目分支提交 PR。默认不允许直接推送 `main`。贡献者可以：

- 提交 bug fix、功能、文档和测试。
- 在 issue 中提出需求、协议变更建议和现场问题。
- 参与 review，但最终合并由维护者执行。

## GitHub 权限建议

如果仓库仍在个人账号下：

- 仓库所有者保留 Admin 权限。
- 长期维护者授予 Maintain 或 Write 权限；只有确实需要改仓库设置的人才给 Admin。
- 临时合作者授予 Triage 或 Write，优先让他们通过 PR 贡献。

如果后续迁移到组织：

- 建议建立 `core-maintainers`、`protocol-reviewers`、`frontend-reviewers`、`ops-reviewers` 四个小组。
- 用 GitHub Rulesets 或 Branch protection 统一保护 `main`。
- 按模块手动或自动请求对应小组审核。

## 分支策略

- `main`：稳定主线，始终应能安装、启动和演示。
- `feature/*`：新增功能。
- `fix/*`：缺陷修复。
- `docs/*`：文档和流程。
- `experiment/*`：探索性原型，默认不承诺合并。
- `release/*`：如需要阶段性交付，可用于冻结演示版本。

不建议长期维护多个偏离很大的开发主线。实验验证后，应拆成小 PR 合并回 `main`。

## 审核等级

### 低风险变更

示例：错别字、README 补充、小型样式调整、非核心脚本注释。

要求：

- 至少 1 位维护者审核。
- 不需要真实设备验证。

### 中风险变更

示例：前端功能、普通后端 API 调整、Fake G1、运行脚本、示例 payload。

要求：

- 至少 1 位维护者审核。
- 通过本地启动或相关脚本验证。
- 若影响用户路径，需要截图或验证说明。

### 高风险变更

示例：设备命令、机器人动作、ACK 语义、Agent 调度、注册表 schema、端口/部署、`third_party` 同步。

要求：

- 至少 2 位维护者或模块审核者审核。
- 对应模块审核者必须参与。
- PR 中写明回滚方式。
- 能用 Fake G1 或无硬件模式验证的，必须先验证。
- 真实设备验证前，需要明确现场安全边界。

## 合并规则

- 所有正式变更必须通过 Pull Request。
- PR 作者不应自行合并自己的高风险变更。
- 审核意见必须处理完，GitHub conversations 必须 resolve。
- 推荐使用 Squash merge，合并信息保留 PR 编号和变更摘要。
- 合并后如发现演示阻断问题，优先 revert 该 PR，再重新修复。

## 发布与演示版本

当项目要用于演示、联调或阶段性交付时，建议：

- 从 `main` 打 tag，例如 `v2026.06.demo-1` 或 `v0.4.0`。
- 在 tag 或 release note 中写明模型配置、端口、设备接入方式、已验证场景。
- 演示前冻结高风险 PR，只允许明确的 bug fix。

## 贡献版本号

所有上传分支或 Pull Request 都应标注贡献版本号，格式见 `README.md` 的“上传版本号标注”。贡献版本号用于协作追踪，正式发布版本由维护者统一打 tag。
