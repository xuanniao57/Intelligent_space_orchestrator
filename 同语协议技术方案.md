## 一、核心叙事

### 1.1 问题定义

**原来冲之场会感知；现在冲之场会反应。不仅是会反应，而且会联动的反应。**

物理空间中的"物种"（传感器、机器人、灯光、扬声器、人）各自使用不同的"语言"：传感器输出模拟信号/二进制帧，机器人使用ROS2 Topic，灯光走DMX协议，扬声器播放音频，人使用自然语言。它们之间没有互操作层——每个跨物种交互都需要人工编写适配代码。

**同语（Common Language）** 的核心不是定义一个新的数据格式，而是建立一个**共享语义空间**——在这个空间中，环境、机器人、灯光/声音、人、场地能够在同一套概念体系下理解彼此的状态、意图和请求。

### 1.2 两个核心概念

| 概念 | 定义 | 技术对应 |
|------|------|----------|
| **同语** | 跨物种共享的语义空间 + 统一消息协议 | 4类消息类型 + 5种协议动词 + 共享语义标签体系 |
| **联动反应** | 单一感知事件触发多个执行器协同响应的因果闭环 | ECA规则引擎 + 多智能体任务分配 + 时序调度 |

### 1.3 与平台/世界模型的关系

```
平台 (Urban-Hermes + CityAI)     →  物理沙箱，提供空间底板、人群画像、多智能体编排
世界模型 (Political-Organization) →  认知框架，空间要素的状态推理和意图生成
同语 (Talking Spaces)             →  对话语法，不同智能体之间的统一通信协议

链式关系：
  在仿真平台中训练世界模型 →
  世界模型学会生成同语消息 →
  同语消息驱动真实空间中的设备联动 →
  Sim-to-Real 通信协议迁移验证
```

**关键洞察**：世界模型是训练同语的"母语环境"——不同空间要素、不同智能体在世界模型中通过同语相互对话，这些对话协议可以直接部署到物理空间。

## 二、同语协议：完整技术规范

### 2.1 协议分层架构

```
┌──────────────────────────────────────────────┐
│ Layer 4: 语义概念层 (Semantic Concepts)        │
│ 共享本体: zone, density, temperature, intent... │
├──────────────────────────────────────────────┤
│ Layer 3: 意图原语层 (Intent Primitives)        │
│ 5种动词: request, invite, alert, negotiate,   │
│          respond                              │
├──────────────────────────────────────────────┤
│ Layer 2: 消息信封层 (Message Envelope)         │
│ 4种消息: SensorFrame, AgentIntention,          │
│          DeviceCommand, NegotiationProposal    │
├──────────────────────────────────────────────┤
│ Layer 1: 传输层 (Transport)                    │
│ HTTP, WebSocket, MQTT, ROS2 Bridge             │
└──────────────────────────────────────────────┘
```

### 2.3 五种协议动词

| 动词 | 方向 | 语义 | 触发条件 | 响应要求 |
|------|------|------|----------|----------|
| `request` | Agent→Agent | 请求服务或资源 | 能力不足时 | 需要respond |
| `invite` | Agent→Agent | 邀请参与或转移 | 访客交接、任务委托 | 需要ack |
| `alert` | Agent→All | 广播警告 | 异常状态检测 | 无需回复 |
| `negotiate` | Agent↔Agent | 双向协商任务分配 | 多智能体协调 | 多轮对话 |
| `respond` | Agent→Agent | 响应之前的request/negotiate | 收到请求后 | 单次回复 |

### 2.4 共享语义标签体系（Semantic Tags）

这是"同语"的核心——所有"物种"共享的概念词汇表：

```yaml
空间语义 (Spatial):
  - zone: garden_01, social_circle_01, parking_01, showcase_01, entrance, path_north, path_south
  - position: center, boundary, corner, entrance_point
  - distance: near (<2m), mid (2-8m), far (>8m)

状态语义 (State):
  - thermal: hot (>30°C), warm (22-30°C), cool (15-22°C), cold (<15°C)
  - moisture: dry (<25%), normal (25-65%), wet (>65%)
  - density: empty (<0.1), sparse (0.1-0.3), moderate (0.3-0.6), crowded (>0.6)
  - noise: silent (<30db), quiet (30-50db), active (50-70db), loud (>70db)

意图语义 (Intent):
  - social: greet, guide, invite_gather, disperse, warn
  - environmental: water_alert, temp_warning, light_adjust
  - safety: evacuate, cordon_off, monitor
  - maintenance: check_status, request_service, report_error

执行语义 (Action):
  - robot: navigate, gesture, speak, patrol, standby
  - speaker: announce, whisper, broadcast, play_soundscape
  - laser: guide_path, warning_pattern, ambient_glow, spotlight
  - projection: display_info, show_warning, ambient_visual
```

## 三、联动反应引擎：ECA规则设计

### 3.1 ECA引擎架构

```
Event (感知事件) → Condition (条件匹配) → Action (联动动作序列)
```

**事件源**：
- `sensor_frame_received` — 冲之场传感器数据到达
- `person_detected` — RGB-D相机检测到人
- `audio_event` — SAN麦克风检测到声音事件
- `agent_intention_received` — 智能体发布意图
- `negotiation_request` — 收到协商请求

### 3.2 规则定义格式

```json
{
  "rule_id": "R001_garden_water_stress",
  "description": "花园植物缺水 → 机器人巡逻 + 扬声器提醒 + 灯光切换",
  "trigger": {
    "event_type": "sensor_frame_received",
    "conditions": [
      {"field": "sensor_values.plant_water_stress", "operator": "gte", "value": 0.6},
      {"field": "sensor_values.people_density", "operator": "gte", "value": 0.3}
    ]
  },
  "actions": [
    {
      "target": "unitree_g1",
      "command_type": "g1.action",
      "params": {
        "action": "navigate_and_speak",
        "waypoints": [{"x": 2.5, "y": 1.8}],
        "speech_text_template": "花园里的植物需要浇水，当前土壤湿度仅{soil_moisture}%",
        "gesture": "point_to_garden"
      },
      "delay_ms": 0,
      "priority": 0.7
    },
    {
      "target": "speaker_gateway",
      "command_type": "speaker.speak",
      "params": {
        "text_template": "检测到花园区域植物缺水，请留意自动浇灌提示",
        "target_nodes": ["SAN-01"],
        "volume": 55
      },
      "delay_ms": 200,
      "priority": 0.5
    },
    {
      "target": "laser_gateway",
      "command_type": "laser.scene",
      "params": {
        "scene": "water_alert",
        "color": "soft_blue",
        "intensity": 0.4,
        "zone": "garden"
      },
      "delay_ms": 100,
      "priority": 0.4
    }
  ],
  "cooldown_ms": 300000
}
```

### 3.3 完整ECA规则库（本次MVP）

| 规则ID | 触发条件 | 联动动作 | 优先级 | 冷却 |
|--------|----------|----------|--------|------|
| R001 | plant_water_stress ≥ 0.6 | G1巡逻+播报, SAN提醒, 灯光蓝 | 0.7 | 5min |
| R002 | people_density ≥ 0.7 AND temperature ≥ 30 | G1疏导, SAN广播, 灯光琥珀色 | 0.9 | 3min |
| R003 | person_detected at entrance | G1迎宾, SAN欢迎, 路径灯光引导 | 0.8 | 1min |
| R004 | noise_db ≥ 75 in garden | SAN提示降噪, G1温和提醒 | 0.6 | 2min |
| R005 | temperature ≥ 35 | SAN高温预警, 灯光红色脉冲 | 0.85 | 5min |

**协议动词**：alert, respond
**端到端延迟**：< 2秒

         situation="plant_water_crisis"

t=200ms  ECA规则R001触发，并行下发3条DeviceCommand:

## 五、平台架构：超越斯坦福小镇

### 5.1 斯坦福小镇范式的问题

Stanford Generative Agents (Park et al., 2023) 是开创性工作，但其设计假设不适用于具身智能世界模型：

| 维度 | 斯坦福小镇 (2023) | 我们的需求 |
|------|-------------------|-----------|
| 空间表示 | 2D网格 (x, y坐标) | 3D场景图 (x, y, z + 语义层) |
| 智能体类型 | 纯文本LLM Agent | 多模态智能体 (视觉/语言/空间/运动) |
| 物理仿真 | 无 (瞬间移动) | 刚体动力学 + 机器人运动学 |
| 设备映射 | 无 (纯虚拟) | 真实传感器/执行器一对一映射 |
| I/O模态 | 文本输入输出 | 29维传感器 + 音频 + 灯光 + 机器人动作 |
| 通信协议 | 无 (Agent间仅通过"观察"交互) | 统一同语协议 (4种消息, 5种动词) |
| 空间角色 | 被动容器 (2D地图) | 主动参与者 (具有Persona/记忆/意图) |
| 实时性 | 无要求 (离散时间步) | 强实时 (<100ms传感器到执行器) |

## 七、关键设计决策

### 7.1 为什么是规则驱动的ECA而非端到端LLM？

**本次MVP选择规则驱动**，理由：
1. 延迟可控：ECA规则匹配 < 10ms，LLM推理 500-2000ms
2. 行为可预测：规则输出确定，适合安全敏感的物理空间
3. 调试可追溯：每条规则有明确ID和触发条件
4. 渐进升级路径：规则可以逐步替换为LLM生成的动态规则

**下次进阶方向**：LLM在规则匹配失败时作为fallback，动态生成新规则

### 7.2 为什么同语协议采用JSON而非二进制？

1. 多传输层兼容：HTTP/WS/MQTT都原生支持JSON
2. 人机可读：调试和演示时可直接阅读消息内容
3. ESP32友好：ArduinoJSON库成熟
4. 扩展性：新增字段不破坏向后兼容

**性能备注**：当前传感器帧 < 500 bytes，机器人命令 < 1KB，在100Mbps局域网中序列化开销可忽略。

### 7.3 为什么现在不直接做世界模型内同语训练？

世界模型内端到端训练通信协议是长期目标，但当前阶段：
- 需要先在物理空间中验证同语协议的语义设计是否合理
- ECA规则库提供了"ground truth"——LLM可以学习这些规则
- Sim-to-real gap需要先收集真实空间的交互数据

**渐进路径**：
1. 阶段1（本次）：规则驱动的ECA → 收集人机交互数据
2. 阶段2（下次）：数据训练LLM → LLM辅助规则生成
3. 阶段3（远期）：仿真中训练 → sim-to-real迁移 → 自主同语
